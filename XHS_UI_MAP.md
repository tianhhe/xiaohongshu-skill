### 评论发送流程（已验证成功 2026-03-03）

**Step 1: 找到评论输入框**
- **核心元素：** `p.content-input[contenteditable="true"]` — 这是真正的输入框
- **不是** `<input>` 或 `<textarea>`，是 `contenteditable` 的 `<p>` 标签
- 初始状态：较窄（约117x40），显示占位文字"说点什么..."在旁边的 `span` 中
- **定位方法：** JS 搜索 `contenteditable="true"` 且 y > 300 的元素
- 常规 CSS 选择器（`[placeholder*="说点什么"]`、`.comment-input` 等）全部匹配不到

**Step 2: 点击激活**
- 点击 `p.content-input` 后，输入区域**展开**：
  - 宽度从 ~117px 变成 ~367px
  - y坐标上移约48px（展开空间）
  - 上方出现 **表情快捷按钮**（😊🤗😎 三个图标）
  - 底部出现 **@ 按钮、😊 表情、"发送"按钮、"取消"按钮**
- 点击后需要再次 JS 搜索 `contenteditable`，因为坐标/尺寸已变

**Step 3: 输入文字**
- 直接用 `keyboard.type()` 逐字输入即可
- contenteditable 元素获得焦点后能正常接收键盘输入

**Step 4: 发送**
- **发送按钮：** `<BUTTON>` 元素，文字为"发送"
- 用JS搜索文字为"发送"的按钮，获取坐标后点击
- 发送成功后：
  - 左侧图片区弹出 **"评论成功"** toast 提示（约2秒消失）
  - 评论立即出现在评论区最顶部
  - 评论数 +1
  - 互动栏恢复为初始状态（输入框缩回）

**DOM 结构（完整层级）：**
```
div.interactions.engage-bar (最外层容器)
  └ div.engage-bar-container
    └ div.engage-bar
      └ div.input-box
        ├ div.content-edit
        │  └ div.not-active.inner-when-not-active
        │    └ div.inner
        │      └ span "说点什么..."
        └ p.content-input [contenteditable="true"]  ← 真正的输入框
      └ button "发送"  ← 激活后才出现
      └ "取消"  ← 激活后才出现
```

**踩坑记录：**
- 用坐标点击(700~830, 870~895)容易点到左侧图片区（触发全屏预览）或弹窗外（关闭弹窗）
- `.input-box` 选择器会匹配到顶部搜索框
- `.engage-bar [class*="input"]` 可能匹配到点赞按钮附近
- **可靠方案：用 JS 的 `querySelectorAll('[contenteditable="true"]')` 定位，再通过 y>300 过滤**
- 有些帖子打开后DOM渲染需要时间，首次JS搜索可能找到0个元素，需要多等1-2秒重试

**关闭方式：** 按 Escape 或点击左上角 ×

**已知点赞状态检测：** 红色(rgb(255,36,66) 或 rgb(255,72,72)) = 已点赞

### 二级回复流程（已验证成功 2026-03-03）

> 二级回复 = 针对某条评论的嵌套回复，区别于一级评论（顶级评论）。

**Step 1: 找到目标评论的"回复"按钮**
- 每条评论下方有 `❤赞` 和 `💬回复` 两个操作按钮
- "回复"按钮是 `<span class="count">` 元素，文字为"回复"
- **定位方法：** JS 搜索所有 `span.count`，筛选 `textContent === '回复'`，再通过父级 `comment` 容器找到对应的评论者昵称
- 回复按钮 x 坐标约 789~801（非作者评论）或 825~837（作者自己的评论，稍偏右）
- **不要硬编码 y 坐标**，因为评论区会滚动，每次都要用 JS 实时获取

**Step 2: 点击"回复"按钮**
- 点击后，底部弹出回复输入框（与一级评论的输入框复用同一位置）
- 输入框上方显示引用信息：**"回复 [用户名]"** + 原评论内容摘要
- 出现三个表情快捷按钮 + @ 按钮 + 😊 表情 + "发送"按钮 + "取消"按钮
- **注意：** 点击"回复"后页面可能会自动滚动，导致评论区位置变化

**Step 3: 输入文字**
- 点击输入框区域（约 x=830, y=610~620）确保焦点
- 用 `keyboard.type()` 逐字输入，与一级评论相同

**Step 4: 发送**
- **发送按钮：** `<BUTTON>` 元素，文字"发送"，固定位置约 (934, 668)
- 用 JS 搜索确认坐标：`document.querySelectorAll('button')` 找文字为"发送"且 y > 300 的
- 发送成功后：
  - 二级回复显示在对应评论下方，缩进展示，带"作者"标签
  - 评论总数 +1
  - 输入框恢复为初始状态

**踩坑记录（重要！）：**
- **坐标点击"回复"容易失败**：直接用截图估算的坐标经常偏移几个像素就点不中，或者点到弹窗外导致弹窗关闭。**必须用 JS 精确获取回复按钮的实时坐标**
- **弹窗关闭问题**：如果点击位置略微偏离评论区（特别是偏左），会点到弹窗外面导致帖子详情关闭，需要重新打开帖子、滚动到评论区重来
- **发送按钮坐标**：视觉上看起来在 y≈640 附近，但实际 JS 获取的中心点是 y≈668。截图估算的坐标不可靠，每次都应该用 JS 确认
- **页面滚动影响**：点击"回复"后，页面有时会自动滚动到顶部，需要重新滚下来。评论区滚动后所有按钮 y 坐标都会变，必须重新 JS 定位
- **一级 vs 二级的区别**：`xn_live.py comment` 命令只能发一级评论（顶级评论），要发二级回复必须手动点击评论下方的"回复"按钮来触发回复模式
- **回复按钮区分**：作者自己回复的评论的"回复"按钮 x 偏右约 837，普通用户评论的"回复"按钮 x 约 801。可以通过父级容器的用户名来区分是哪条评论的回复按钮

**推荐操作流程（最可靠）：**
```
1. JS 获取所有 span.count 中文字为"回复"的元素坐标 + 父级评论者昵称
2. 根据昵称匹配目标评论，取对应的回复按钮坐标
3. human_click 点击回复按钮
4. 等待 1-2 秒，截图确认"回复 XXX"输入框已出现
5. 点击输入框区域确保焦点
6. keyboard.type() 逐字输入
7. JS 获取发送按钮精确坐标
8. human_click 发送按钮
9. 等待 2 秒，截图确认发送成功
```
